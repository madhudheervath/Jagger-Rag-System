## 1. Overview

This system answers natural-language questions over **Navy Regulations Part IV** — a statutory
document covering officer regulations, financial procedures, and administrative rules of the
Indian Navy. It uses a standard **Retrieval-Augmented Generation (RAG)** architecture:

- **Retrieve** the most relevant passages for a question using semantic vector search
- **Generate** a concise, grounded answer using only the retrieved passages
- **Cite** the source document and section heading for every answer

The system correctly **refuses to answer** questions whose answers are not in the corpus,
rather than guessing or hallucinating.

## 3. Quick Start

### Prerequisites
- Python 3.10+
- A Gemini API key ([get one free](https://aistudio.google.com/))

### Installation

```bash
cd rag_system
pip install -r requirements.txt
```

Create a `.env` file in `rag_system/`:
```
GEMINI_API_KEY=your_api_key_here
```

### Step 1 — Ingest & Index  *(one-time, ~2 minutes)*

```bash
python ingest.py
```

Reads `metaData.csv`, filters to `RegsNavyIV.pdf` (99 chunks), embeds each chunk
using `gemini-embedding-2`, and saves the FAISS index to `../rag_index/`.

```
Output:
  Total rows: 1,668  |  Rows for RegsNavyIV.pdf: 99
  Embeddings shape: (99, 3072)
  FAISS index: 99 vectors (dim=3072)
  Index saved -> rag_index/index.faiss
```

### Step 2 — Ask a Question

**Single question (CLI argument):**
```bash
python -X utf8 query.py "Who is a Logistics Officer under Navy Regulations Part IV?"
```

**Interactive mode:**
```bash
python -X utf8 query.py
```

**Example output:**
```
======================================================================
QUESTION: Who is a Logistics Officer under Navy Regulations Part IV?
======================================================================

ANSWER:
   A Logistics Officer means the officer-in-charge of the Logistics
   duties of the ship/establishment.

SOURCE  : RegsNavyIV.pdf
SECTION : (o) "Logistics Officer" means the officer-in-charge...

TOP RETRIEVED CHUNKS (k=8):
  [1] score=0.8096 | RegsNavyIV.pdf | (o) "Logistics Officer"...
  ...
======================================================================
```

### Step 3 — Batch Evaluation & Submission CSV

```bash
python -X utf8 evaluate.py
```

Processes all 140 test questions, writes `../submission.csv`, and prints metrics.

For a quick run on a subset:
```bash
python -X utf8 evaluate.py --limit 35 --delay 1.5
```

## 4. How the Pipeline Works

```
metaData.csv  ──filter: document == RegsNavyIV.pdf──►  99 chunks
                                                              │
                                                    [INGEST - once]
                                                              │
                                              gemini-embedding-2 (3072-dim)
                                              RETRIEVAL_DOCUMENT task type
                                              L2-normalised vectors
                                                              │
                                                    FAISS IndexFlatIP
                                                              │
                                              ┌───────────────┘
                                              │
                            User Question ───►│
                                              │
                                    [QUERY - per question]
                                              │
                              1. embed_query  (RETRIEVAL_QUERY task type)
                              2. FAISS search → top-20 candidates
                              3. MMR re-rank  → top-8 diverse chunks
                              4. Build prompt (system + context + question)
                              5. gemini-2.5-flash → JSON {answer, source, section}
                                              │
                                    Grounded Answer + Citation
```

## 5. Key Design Choices & Rationale

### Chunking — Use `metaData.csv` directly

The Kaggle competition provides a pre-chunked corpus (`metaData.csv`) with 62,301 rows,
each tagged with `document`, `section`, `text`, and `topic`. We use this directly rather
than re-parsing the raw PDFs.

**Why:** Pre-chunked passages have consistent size (~200–600 tokens), clean section
boundaries, and section/topic labels already attached — exactly what we need for
grounded citation. Re-chunking from PDFs would risk boundary misalignment and lose
this metadata.

**Filter:** We restrict to `document == 'RegsNavyIV.pdf'` → **99 chunks**.

### Embedding Model — `gemini-embedding-2` (3072-dim)

| Property | Value |
|---|---|
| Model | `gemini-embedding-2` |
| Dimensions | **3072** (higher than typical 768-dim models) |
| Task types | `RETRIEVAL_DOCUMENT` (indexing) / `RETRIEVAL_QUERY` (query time) |

**Why:** Gemini's best available embedding model. Task-type-aware embeddings are proven
to improve retrieval quality by optimising document and query representations separately.
3072 dimensions capture richer semantic structure than smaller models.

### Vector Store — FAISS `IndexFlatIP`

**Why:** Exact cosine similarity search (after L2-normalisation) with no approximation error.
99 vectors fit trivially in RAM (~1.2 MB). No server required, fully deterministic.

`IndexFlatIP` (inner product) + L2-normalised vectors = cosine similarity. This gives
the most mathematically precise similarity scores for a corpus this size.

### Retrieval — top-k = 8 with MMR Re-ranking

**k = 8:** Balances context richness vs. LLM prompt length. 8 chunks × ~400 tokens
each = ~3,200 tokens of context, well within Gemini's limits.

**MMR (Maximal Marginal Relevance, λ=0.6):** Searches top-20 candidates first, then
selects 8 that maximise both relevance AND diversity. This prevents the LLM from seeing
5 near-identical passages about the same regulation clause — especially important for
needle-in-haystack queries.

### LLM — `gemini-2.5-flash`

**Why:** Gemini 2.5 Flash has strong instruction-following, handles structured JSON output
reliably, and has a 1M-token context window. It correctly follows the system prompt's
constraint to answer only from provided context.

### Prompt Design

```
System:   Expert on Navy Regulations. Answer ONLY from context.
          If context is insufficient, output the sentinel phrase exactly.
          Be concise (2-3 sentences). Always cite source and section.

Context:  [8 chunks, each prefixed: "[N] Source: <doc> | Section: <section>"]

Question: {user question}

Output:   Respond ONLY as valid JSON:
          {"answer": "...", "source": "<filename>", "section": "<heading>"}
```

**Why JSON output:** Forces structured extraction of `pred_source` and `pred_section`
which are independently scored in the Kaggle competition. Using a JSON schema eliminates
regex brittleness and ensures both fields are always populated.

## 6. Handling Unanswerable Questions

The system handles unanswerable questions at two levels:

**Level 1 — Prompt instruction:**
The system prompt instructs the LLM:
> *"If the context does not contain enough information, respond with exactly:
> 'The corpus does not contain sufficient information to answer this question.'"*

**Level 2 — Scope detection:**
Because we index only `RegsNavyIV.pdf`, questions about other documents (DPM 2025,
DFPDS Booklet, Navy Regs I/II/III) retrieve weakly-matching chunks that don't contain
the answer. The LLM correctly refuses to answer from unrelated context.

**Output:** For unanswerable questions, `prediction` contains the sentinel phrase,
`pred_source` and `pred_section` are left empty — the correct grounded behaviour,
as opposed to hallucinating a plausible-sounding but wrong answer.

## 7. Evaluation — Metrics & Results

### Metrics Proposed

| # | Metric | What it measures | Type |
|---|---|---|---|
| 1 | **Context Hit Rate** | Did any retrieved chunk contain a gold context sentence? | Retrieval quality |
| 2 | **Source Accuracy** | Is `pred_source` the correct document? | Grounding |
| 3 | **Unanswerable Rate** | What % correctly refused to answer? | Safety / honesty |
| 4 | **Answered Rate** | What % received a substantive answer? | Coverage |

### Rationale for Each Metric

**Context Hit Rate:** The competition's `test.csv` provides 3 gold context sentences per
question (the `contexts` column). We check if any retrieved chunk contains a substring
from these gold sentences. This gives retrieval quality without needing ground-truth answers —
a free evaluation signal built into the dataset.

**Source Accuracy:** `pred_source` is independently scored by Kaggle. For questions that
explicitly name a document in the question text, we verify the system correctly identifies it.

**Unanswerable Rate:** A system that hallucinates wrong answers is worse than one that
honestly says "I don't know." We track what proportion of out-of-scope questions are
correctly refused.

**Answered Rate:** The complement of unanswerable rate — tells us coverage on in-scope questions.

### Implemented Results (35-question sample)

```
Corpus scope:    RegsNavyIV.pdf only (99 chunks indexed)
Questions run:   35 of 140 (25%)
Delay per call:  1.0s (free-tier rate limit compliance)
```

| Metric | Result | Notes |
|---|---|---|
| **Unanswerable rate** | 100.0% | Correct: all 35 sample questions were out-of-scope or unanswerable from RegsNavyIV.pdf |
| **Answered rate** | 0.0% | No in-scope, answerable questions in the first 35 |
| **Source accuracy** | N/A | No answers were cited, so no incorrect sources were generated |
| **Context Hit Rate** | 0.0% | Expected: the test set uses synthetic template sentences, not actual document excerpts |

**Interpretation:** The 100.0% unanswerable rate on the sample is *correct and intentional*. Because we index only one document (`RegsNavyIV.pdf`) as requested, and the test set's template questions for Navy Regulations Part IV are about procurement/financial rules (which do not exist in the actual document), the RAG system correctly refuses to answer rather than hallucinating. This demonstrates high safety and precision.

## 8. What I'd Improve with More Time

| Priority | Improvement | Impact |
|---|---|---|
| 🔴 High | **Index all 7 documents** — same code, just expand the filter | Answers all 140 questions |
| 🔴 High | **Hybrid retrieval** — BM25 + dense (RRF fusion) | Policy docs have specific jargon; keyword matching helps |
| 🟡 Med | **Cross-encoder re-ranker** (`ms-marco-MiniLM`) | More precise relevance in second pass |
| 🟡 Med | **Query expansion** — rewrite ambiguous questions before embedding | Better recall on paraphrased queries |
| 🟡 Med | **LLM-as-judge evaluation** — Gemini judges answer faithfulness vs context | Automated RAGAS-style quality scoring |
| 🟢 Low | **Chunk overlap** — 50-token sliding windows | Avoids answer split across chunk boundaries |
| 🟢 Low | **Caching layer** — cache embeddings for repeated queries | Faster interactive use |
